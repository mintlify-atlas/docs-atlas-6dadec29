## PagePool: 40% Speedup

### How It Works

Creating and closing a Playwright browser page for each URL has ~200ms overhead. **PagePool** pre-warms a pool of pages and reuses them via `asyncio.Queue`.

```python
class PagePool:
    """Pool of reusable Playwright pages backed by asyncio.Queue.
    
    Avoids the overhead of creating/closing a new page per URL.
    Pages are reset (about:blank + clear cookies) before re-use.
    """
    
    async def acquire(self) -> AsyncGenerator[Page, None]:
        page = await self._queue.get()
        try:
            # Reset state between uses
            await page.goto("about:blank", timeout=5000)
            await page.context.clear_cookies()
            yield page
        finally:
            await self._queue.put(page)
```

See `src/scraper/page.py:242` for the full implementation.

### Configuration

**PAGE_POOL_SIZE** controls the pool size (default: 5):

```bash
docker run -e PAGE_POOL_SIZE=10 docrawl
```

| Value | Behavior | Use Case |
|-------|----------|----------|
| `0` | Disabled (legacy mode) | Debugging or low-memory environments |
| `3-5` | Default | Balanced performance/memory |
| `10-15` | High throughput | Powerful servers with 16GB+ RAM |
| `20+` | Maximum speed | Dedicated crawling infrastructure |

<Warning>
  Each page consumes ~200MB RAM. Set `PAGE_POOL_SIZE * 200MB < available_memory`.
</Warning>

|------|-------|--------------|---------------|----------|

## 3-Tier Cleanup Classification: 30% Token Reduction

### How It Works

Not all content needs LLM cleanup. **3-tier classification** skips code-heavy chunks and short clean text:

| Level | Criteria | Action |
|-------|----------|--------|
| `skip` | Code density &gt;60% OR short (&lt;2000 chars) without noise | Pass through unchanged |
| `cleanup` | Has noise indicators OR long text | Standard LLM cleanup |
| `heavy` | Broken tables OR LaTeX expressions | Extended cleanup with repair |

```python
def classify_chunk(markdown: str) -> CleanupLevel:
    """Classify a chunk by the level of LLM cleanup needed."""
    lower = markdown.lower()
    has_noise = any(indicator in lower for indicator in _NOISE_INDICATORS)
    
    # Mostly-code chunks are clean
    if _code_density(markdown) > 0.6:
        return "skip"
    
    # Short clean text without noise
    if len(markdown) < 2000 and not has_noise:
        return "skip"
    
    # Heavy cleanup for complex content
    if _has_broken_tables(markdown) or _has_latex(markdown):
        return "heavy"
    
    return "cleanup" if has_noise or len(markdown) >= 2000 else "skip"
```

See `src/llm/cleanup.py:112` and [ADR-003](https://github.com/plater7/docrawl/blob/main/docs/adr/ADR-003-three-tier-cleanup-classification.md).

### Noise Indicators

16 patterns trigger cleanup:

- `cookie`, `privacy policy`, `terms of service`
- `subscribe`, `toggle dark`, `light mode`
- `skip to content`, `table of contents`
- `on this page`, `all rights reserved`
- `powered by`

### Benchmarks

| FastAPI docs | 1,200 | 420 (35%) | 720 | 60 | **32%** |
| Python docs | 2,500 | 950 (38%) | 1,400 | 150 | **35%** |
| Stripe API | 800 | 280 (35%) | 480 | 40 | **30%** |

## HTTP Fast-Path: Bypass Playwright for Static Sites

### How It Works

Many documentation sites are **server-rendered HTML** that don't need JavaScript execution. The HTTP fast-path tries plain `httpx` before launching Playwright:

```python
async def fetch_html_fast(url: str) -> str | None:
    """Try to fetch and convert a page to markdown without Playwright.
    
    Returns markdown only if it meets minimum quality threshold (≥500 chars).
    """
    async with httpx.AsyncClient(timeout=8.0) as client:
        resp = await client.get(url)
        if resp.status_code != 200:
            return None
        
        markdown = markdownify(resp.text, strip=["script", "style", "nav", "footer"])
        if len(markdown) >= 500:
            return markdown
    return None
```

See `src/scraper/page.py:15` for the implementation.

### When It Works

<Check>Static HTML sites (Jekyll, Hugo, Docusaurus with SSR)</Check>
<Check>Server-rendered frameworks (Next.js SSR, Rails)</Check>

### When It Fails

<Warning>Client-rendered SPAs (React, Vue without SSR)</Warning>
<Warning>Sites with client-side auth or paywalls</Warning>
<Warning>Infinite scroll or lazy-loaded content</Warning>

### Configuration

Enabled by default. Disable with:

```json
{
  "use_http_fast_path": false
}
```

### Benchmarks

| Docusaurus sites | 85% | 1.5s | 4.2s | **2.8x** |

## Concurrent Page Processing

### max_concurrent Setting

Control how many pages are scraped simultaneously:

```json
{
  "max_concurrent": 5
}
```

| Value | Behavior | Use Case |
|-------|----------|----------|
| `1` | Sequential | Rate-limited APIs, debugging |
| `3` | Default | Balanced |
| `5-10` | High throughput | Fast servers, robust target sites |
| `10+` | Maximum speed | Risk of overwhelming target or triggering rate limits |

<Warning>
  **High concurrency risks**:
  - Target site may block your IP
  - Increased memory usage (each page = ~200MB)
  - Violates `robots.txt` crawl-delay
</Warning>

### How It Works

Semaphore-based concurrency control:

```python
semaphore = asyncio.Semaphore(job_request.max_concurrent)

async def scrape_with_limit(url: str):
    async with semaphore:
        return await scrape_page(url)

await asyncio.gather(*[scrape_with_limit(url) for url in urls])
```

See `src/jobs/runner.py` for the full implementation.

## Page Caching: 24-Hour TTL

### How It Works

Opt-in disk cache stores scrape results with SHA-256 content hashing:

```python
# Enable caching
{
  "use_cache": true
}
```

On subsequent crawls:
1. SHA-256 hash of page content is compared
2. If unchanged, cached result is returned (skips Playwright + LLM)
3. If changed, full re-scrape is performed

### When to Enable

<Check>Development/testing on same site repeatedly</Check>
<Check>Incremental crawls of large sites</Check>
<Check>Sites that rarely change (versioned docs)</Check>

### Cache Storage

In-memory dict (lost on container restart). Cache location:

```
./data/.page_cache/{job_id}/{url_hash}.html
```

<Warning>
  Cache is **NOT persisted across container restarts**. Planned: Redis backend for production.
</Warning>

See `src/scraper/cache.py` for implementation.

## Dynamic Timeouts: Prevent Hangs and Failures

### How It Works

Timeout scales with content size:

```python
def _calculate_timeout(content: str) -> int:
    """Calculate dynamic timeout based on chunk size and token estimate."""
    tokens = _estimate_tokens(content)
    timeout = int(BASE_TIMEOUT + (tokens / 250) * 10)
    return min(timeout, MAX_TIMEOUT)
```

| Content Size | Tokens | Timeout |
|--------------|--------|----------|
| 1KB | ~250 | 45s (base) |
| 5KB | ~1,250 | 95s → **90s** (capped) |
| 10KB | ~2,500 | 145s → **90s** (capped) |

Constants:
- `BASE_TIMEOUT = 45` seconds
- `TIMEOUT_PER_KB = 10` seconds
- `MAX_TIMEOUT = 90` seconds

## Model Selection: Right-Size Your LLMs

<Warning>
  **Critical**: Oversized models cause severe performance degradation. A 70B model for URL filtering can be **10x slower** than a 7B model with no quality gain.
</Warning>

### Recommended Sizing

| Role | Task Complexity | Recommended Size | Examples |
|------|----------------|------------------|----------|
| **crawl_model** | Low (binary classification) | 7B | `mistral:7b`, `qwen2.5:7b` |
| **pipeline_model** | High (markdown cleanup) | 14B-32B | `qwen2.5:14b`, `command-r:35b` |
| **reasoning_model** | Medium (URL filtering) | 7B-14B | `deepseek-r1:7b`, `mistral:7b` |

### Multi-Provider Strategy

Use **different providers for different roles**:

```json
{
  "crawl_model": "ollama/mistral:7b",
  "pipeline_model": "openrouter/anthropic/claude-3.5-sonnet",
  "reasoning_model": "ollama/qwen2.5:7b"
}
```

Benefits:
- Fast local models for mechanical tasks (filtering, classification)
- Powerful cloud models for complex cleanup
- Hybrid cost optimization

## Bottleneck Identification

### Slow Discovery Phase

<Warning>Symptom: Job stuck in discovery for 5+ minutes</Warning>

**Causes**:
- No sitemap available → falls back to recursive crawl
- Large site with deep navigation hierarchy
- LLM filtering enabled on thousands of URLs

**Solutions**:
- Set `max_depth=2` to limit recursion
- Disable LLM filtering for known-good domains
- Use `filter_sitemap_by_path=true`

### Slow Scraping Phase

<Warning>Symptom: &lt;10 pages/minute throughput</Warning>

**Causes**:
- PagePool disabled or too small
- `max_concurrent=1` (sequential)
- All pages require Playwright (no HTTP fast-path hits)
- Target site has rate limiting or slow responses

**Solutions**:
- Increase `PAGE_POOL_SIZE` to 10-15
- Increase `max_concurrent` to 5-8
- Add `delay_ms` to respect rate limits
- Check `docker stats` for memory pressure

### Slow Cleanup Phase

<Warning>Symptom: Each chunk takes 10+ seconds to process</Warning>

**Causes**:
- Oversized LLM model (70B for simple cleanup)
- Ollama not using GPU acceleration
- Small `num_ctx` causing input truncation and retries
- All chunks classified as `heavy` cleanup

**Solutions**:
- Use 7B-14B model for `pipeline_model`
- Enable GPU layers: `OLLAMA_GPU_LAYERS=35`
- Check classification: most chunks should be `skip` or `cleanup`
- Verify CUDA/ROCm drivers are working
