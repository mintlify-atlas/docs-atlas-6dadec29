## Overview

Docrawl uses API key authentication to secure access to its endpoints. Authentication is implemented via the `ApiKeyMiddleware` in the FastAPI application and validates requests using the `X-Api-Key` header.

## Configuration

Authentication is controlled by the `API_KEY` environment variable:

<CodeGroup>
```bash Docker
docker run -e API_KEY=your_secret_key_here docrawl
```

```bash Environment Variable
export API_KEY=your_secret_key_here
```

```yaml Docker Compose
services:
  docrawl:
    image: docrawl
    environment:
      - API_KEY=your_secret_key_here
```
</CodeGroup>

<Note>
If `API_KEY` is not set or is empty, authentication is **disabled** and all endpoints are publicly accessible.
</Note>

## Making Authenticated Requests

Include your API key in the `X-Api-Key` header with every request:

<CodeGroup>
```bash cURL
curl -X POST https://your-docrawl-instance.com/api/jobs \
  -H "X-Api-Key: your_secret_key_here" \
  -H "Content-Type: application/json" \
  -d '{
    "start_url": "https://docs.example.com",
    "model": "qwen3-coder:free"
  }'
```

```javascript JavaScript
const response = await fetch('https://your-docrawl-instance.com/api/jobs', {
  method: 'POST',
  headers: {
    'X-Api-Key': 'your_secret_key_here',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    start_url: 'https://docs.example.com',
    model: 'qwen3-coder:free'
  })
});
```

```python Python
import requests

response = requests.post(
    'https://your-docrawl-instance.com/api/jobs',
    headers={
        'X-Api-Key': 'your_secret_key_here',
        'Content-Type': 'application/json'
    },
    json={
        'start_url': 'https://docs.example.com',
        'model': 'qwen3-coder:free'
    }
)
```
</CodeGroup>

## Authentication-Exempt Endpoints

The following endpoints do **not** require authentication, even when `API_KEY` is configured:

| Endpoint | Description |
|----------|-------------|
| `/` | Web UI homepage |
| `/api/health/ready` | Health check and readiness status |

<Note>
These endpoints are exempt to allow health checks and UI access without authentication.
</Note>

## Error Responses

### 401 Unauthorized

Returned when the API key is missing or invalid:

```json
{
  "detail": "Unauthorized — X-Api-Key required"
}
```

<Warning>
**Common causes:**
- Missing `X-Api-Key` header
- Incorrect API key value
- Trailing whitespace in the API key
</Warning>

## Security Best Practices

<CodeGroup>
```bash Generate Strong Key
# Generate a secure random API key (Linux/macOS)
openssl rand -hex 32

# Or use Python
python3 -c "import secrets; print(secrets.token_hex(32))"
```
</CodeGroup>

<Warning>
**Security recommendations:**
- Use a strong, randomly generated API key (at least 32 characters)
- Store the API key securely (use environment variables, never commit to version control)
- Rotate keys periodically
- Use HTTPS in production to prevent key interception
- Consider using different keys for different environments (dev, staging, production)
</Warning>

## Implementation Details

The authentication middleware is defined in `src/main.py:172-187`:

```python
class ApiKeyMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        if not _API_KEY:
            return await call_next(request)
        if request.url.path in _AUTH_EXEMPT:
            return await call_next(request)
        key = request.headers.get("X-Api-Key", "")
        if key != _API_KEY:
            return JSONResponse(
                status_code=401,
                content={"detail": "Unauthorized — X-Api-Key required"},
            )
        return await call_next(request)
```

The middleware performs the following checks in order:
1. If no API key is configured, allow all requests
2. If the path is in the exempt list, allow the request
3. Validate the `X-Api-Key` header matches the configured key
4. Return 401 if validation fails
